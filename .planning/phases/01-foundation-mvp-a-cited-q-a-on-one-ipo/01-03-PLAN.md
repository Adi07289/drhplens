<objective>
Build the offline ingestion pipeline that turns the committed Swiggy DRHP PDF into a searchable Qdrant Cloud collection. This is build-time work (the HF Spaces app never re-indexes — PITFALL Pitfall 1). The pipeline establishes the storage-bus invariant (ARCHITECTURE Pattern 5): batch pipelines write; the on-demand agent in Wave 3 only reads. This plan also commits the parsed Docling JSON to the repo so embedding-model changes in future phases don't require re-parsing (ARCHITECTURE Anti-Pattern 4 mitigation).

Purpose: INGEST-02 (parse 300-500 page DRHPs incl. tables + risks + promoter) and INGEST-03 (section-aware chunking with valid payload metadata) ship here. Phase 2's multi-IPO catalogue inherits the chunking + embedding contract unchanged.

This plan has ONE checkpoint:human-verify task — the Docling parse output must be eyeballed once (PITFALL 4 detection: "page index 0 of the Docling output should be the cover page, not 'page 1 of Risk Factors'"). Pure automation can't catch a pagination-drift bug because both schemas are syntactically valid; only a human looking at the cover-page text vs page index 0 can confirm.

Output:
- Parsed `swiggy_prospectus_2024_11.docling.json` committed to repo
- Qdrant Cloud `drhp_chunks` collection populated with 1500-2500 chunks
- `tools/embedder.py` + `tools/reranker.py` wrappers (`@st.cache_resource` singletons; CPU-friendly)
- `storage/vector.py` with `ChunkPayload` TypedDict matching `agent/schemas.RetrievedChunkRef` field names
- Wave 0 stubs `test_parser.py`, `test_chunker.py`, `test_embedder.py`, `test_qdrant_ingest.py` flipped to GREEN
</objective>
