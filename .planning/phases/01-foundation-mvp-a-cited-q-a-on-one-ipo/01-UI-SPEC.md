# Phase 1 — UI Design Contract: Foundation + MVP-A (Cited Q&A on One IPO)

> Visual and interaction contract for the first publicly-deployed surface of DRHPLens. Generated by gsd-ui-researcher; consumed by gsd-planner, gsd-executor, gsd-ui-checker, and gsd-ui-auditor. Every prescriptive value below is locked unless the checker flags a regression.

---

## Audience & Posture

Two simultaneous audiences shape every decision:

1. **Primary — Indian retail investor on mobile.** Evaluating an IPO (target: Swiggy Nov 2024) they may already have applied for. Wants signal not hype; has been burned by hallucinating finance chatbots; mobile-first.
2. **Secondary — DS / ML engineering recruiter.** Lands via resume deep-link, often on `/methodology`. Reads first-screen quality as a proxy for engineering judgement.

**Posture in three words:** **Honest. Cited. Calm.** No green/red, no exclamation marks, no shouting. Uncertainty is a feature surfaced visibly — never buried — because Phase 5's calibrated forecast (UI-03) will need this precedent.

## Spacing Scale

Declared values (multiples of 4, 8-point scale). Applied via Streamlit's default vertical rhythm + CSS-injected margins on custom HTML blocks.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gap between citation-chip cluster and surrounding text |
| sm | 8px | Padding inside citation chips; gap between disclaimer footer line height |
| md | 16px | Default vertical gap between paragraphs in an answer; padding inside reformulation-suggestion chips |
| lg | 24px | Section padding inside the answer block; gap between DRHP metadata header and the question input |
| xl | 32px | Layout gap between major regions (header → chat → footer) on desktop |
| 2xl | 48px | Top padding above the first answer of a session |
| 3xl | 64px | Modal vertical padding on desktop (collapses to `lg` on mobile <640px) |

**Exceptions:**
- **Touch targets:** Citation chips and reformulation-suggestion chips must have a minimum interactive surface of **44×44px** on mobile (Apple HIG / WCAG 2.5.5 AAA). Visual chip can be smaller (~28×20px) — the tap target is enlarged via transparent padding.
- **Persistent footer minimum height:** 56px on mobile, 48px on desktop, to accommodate the wrapped disclaimer at 12px / line-height 1.5 without truncation.

## Typography

**Four sizes, two weights.** All px values in CSS reference rems via Streamlit defaults — values below are the resolved px on a default-zoom desktop and represent the contract.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Small | 12px | 400 (regular) — italic variant for per-answer disclaimer | 1.5 | Persistent disclaimer footer (SEBI 10pt-equivalent floor; 12px ≈ 10.5pt on 1×, comfortably above the regulator minimum) AND per-answer disclaimer "Informational only — not advice" (italic variant). Both legal/contextual roles share this size — 1px hierarchy steps add no meaningful information. |
| Body | 16px | 400 (regular) — 600 (semibold) variant for emphasis | 1.55 | Answer prose; refusal copy; modal body; methodology stub prose; DRHP metadata header label-value pairs (label uses semibold variant); reformulation suggestion chip labels (semibold for finger-friendly visual weight on tap targets). |
| Heading | 20px | 600 (semibold) | 1.3 | Page heading ("Ask DRHPLens about Swiggy"); modal heading; per-section markers; empty-state heading; refusal-banner heading. |
| Display | 28px | 600 (semibold) | 1.2 | Home page hero line + `/methodology` stub heading. Used at most once per page. |

**Derived (not a discrete scale size):** The citation chip number renders via `font-size: 0.7em` and `vertical-align: super` relative to the surrounding 16px body — this is implementation CSS for an inline annotation, not a standalone typographic role. See the Citation Chip Contract in Visuals for the full chip rendering spec.

**Rules:**
- Body line-height of 1.55 protects citation-chip baseline: even with superscript chips embedded mid-sentence, the line rhythm doesn't break because the chip number is rendered at 0.7em with `vertical-align: super` and `line-height: 1.0`.
- No font sizes outside the four declared roles. No italic except the per-answer disclaimer (Small italic variant). No underline except the hover state on citation chips (see Visuals).
- The DRHP metadata header avoids tabular-nums in Phase 1 — full Indian-context numeric formatting is Phase 4 (UI-04). However, issue-size in the metadata header MAY use lakh/crore notation already if it's available in the hand-curated metadata block (low-effort win, matches the user's mental model).

## Visuals — Citation Chip Contract

Citation chips are the load-bearing UI element of Phase 1. Their visual contract:

**Render (in prose):**
```
The DRHP discloses that promoters hold 31.4% post-issue.[1] Related-party
transactions total ₹X cr in FY24, primarily with subsidiaries.[2][3]
```

**HTML structure (injected via `st.markdown(unsafe_allow_html=True)`):**
```html
<sup class="drhp-cite" data-claim-id="c_4f3a"
     tabindex="0" role="button" aria-describedby="cite-1-source">[1]</sup>
```

**Visual properties:**
- Size: 11px font, semibold, white text on `#1E40AF` background
- Shape: pill — `padding: 1px 6px; border-radius: 8px`
- Position: superscript via `vertical-align: super` and `font-size: 0.7em` relative to 16px body
- Cluster: deduped at the surface — three consecutive sentences citing the same span render one chip; underlying `claim_id` traces still record each individual claim-source mapping (so Phase 3's METHOD-01 pane sees the full data)
- Tap target: 44×44px minimum on mobile via transparent padding (`::before` pseudo-element with absolute positioning enlarges the hit area without changing the visible chip)

**States:**
- **Default:** background `#1E40AF`, white text, no border
- **Hover (desktop):** background `#1E3A8A` (one step darker), underline appears on the bracketed number
- **Focus (keyboard tab):** 2px outline ring in `#1E40AF` with 2px offset; visible at any zoom level
- **Active (clicked, expander open):** background `#1E3A8A`, the corresponding `st.expander` below the answer opens to show the source-text snippet
- **Visited:** No visited state — chips are session-scoped, not navigational links

**Expander reveal content (per chip):**
- DRHP source-text snippet, verbatim, in 14px regular (label size), italic, max 4 lines (rest scrollable inside the expander)
- One link: `View DRHP page {N} on SEBI →` opening in a new tab to the SEBI/BSE/NSE-hosted PDF at the cited page anchor (`#page={N}` if PDF.js-compatible; otherwise scroll-to-top with a "search for: {first 5 words}" hint in the expander label)
- Footer line in 12px muted: `claim_id: c_4f3a · DRHP page {N} · section "{section name}"` — this metadata exposure is the precedent Phase 3 METHOD-01 will lean on; keep it visible from day one

**Visualization rules:**
- Citation chips never break a line. If the chip would land at a line break, CSS `white-space: nowrap` keeps the preceding word + chip together.
- A sentence ending with a chip places the period BEFORE the chip: `...promoters hold 31.4% post-issue.[1]` (Perplexity convention; matches academic citation pattern).
- Chips render only when the cite-check node has validated the underlying claim. If cite-check fails for a claim, the whole answer is blocked and the refusal banner shows — never render an "unsourced" chip with no expander content.

## Visuals — Refusal Banner Contract

When the dual-gate refuses (retrieval-floor below threshold OR cite-check finds an unsupported claim), the user sees:

**Visual structure:**
- Background: `#FEF3C7` (warm amber 50)
- Left border: 4px solid `#D97706` (amber 600)
- Text color: `#78350F` (amber 900)
- Padding: `lg` (24px) vertically, `md` (16px) horizontally
- Border-radius: `8px`
- Width: matches the answer block (max 720px on desktop, full-bleed on mobile)
- Position: where the answer would have rendered — same place the user expected an answer, no relocation

**Content layout (top to bottom):**
1. Refusal heading line (heading 20px / 600): `This isn't in the DRHP.` — or, for multi-part partial-grounding: `The DRHP only addresses part of this.`
2. Refusal body (body 16px / 400): the templated refusal copy from the Copywriting Contract above, with the topic and suggestions interpolated
3. Reformulation suggestion chips (label 14px / 600): up to three clickable chips rendered as `Surface-secondary` background pills with `Text-primary` text and `Border-subtle` 1px border. Clicking a chip fills the question-input box with the suggested question (does NOT auto-submit — user retains agency)
4. Per-answer disclaimer (Small 12px italic muted): the same `Informational only — not advice.` footer that appears below grounded answers

**What NOT to do (anti-patterns):**
- Do not use red. Red implies error or threat; a refusal is honest, not punitive.
- Do not use exclamation marks. "This isn't in the DRHP!" reads as accusatory.
- Do not collapse the refusal banner into a `st.warning` toast — refusals are first-class output, not chrome.
- Do not hide the reformulation chips behind an expander — the suggestion IS the value.

## Visuals — Disclaimer Surfaces

Three surfaces, one source of truth (the `DisclaimerSurface` abstraction noted in CONTEXT code_context):

### 1. First-use modal (`st.dialog`)

- Trigger: first session per browser (gated by `st.session_state` + a localStorage-equivalent persistent flag set inside the modal-accept callback so it survives a page reload)
- Backdrop: 60% black overlay
- Dialog size: `min(640px, 90vw)`; vertical centered
- Padding: `3xl` (64px) on desktop, `lg` (24px) on mobile (<640px)
- Heading: `Read this once.` — Heading 20px / 600
- Body: full anchor disclaimer copy (body 16px / 400), 3-4 lines maximum on desktop
- CTA: single primary button `I understand — open DRHPLens` (accent fill, white text, 16px semibold, `md`-padding, `8px` radius). No secondary "Decline" button — declining means leaving the site.
- No close X icon in Phase 1. Acceptance is intentional, single-step.

### 2. Persistent slim footer (every page)

- Fixed at bottom of viewport on mobile (`position: sticky; bottom: 0`); fixed at bottom of page content on desktop
- Background: `#FFFFFF` with `border-top: 1px solid #E2E8F0`
- Height: minimum 56px on mobile, 48px on desktop (allows two lines on mobile, one line on desktop)
- Padding: `sm` (8px) vertical, `md` (16px) horizontal
- Text: 12px regular `#475569` (muted)
- Content: full anchor copy + " · methodology" suffix where "methodology" is an accent-color link to `/methodology`
- Z-index: 50 — above answer content but below the first-use modal (z-index 100)

### 3. Per-answer footer (below every generated answer)

- Renders inside the answer block, after the last citation-chip expander
- Margin-top: `md` (16px) from the answer body
- Separator: `1px` top border `#E2E8F0`, padding-top `sm` (8px) inside the border
- Text: Small 12px italic `#475569`
- Content: `Informational only — not advice.`
- Does NOT render on the refusal banner — the refusal banner has its own per-answer footer baked into its structure

**SEBI 10pt-equivalent verification:** 12px at default browser zoom = ~10.5pt — comfortably above the SEBI Jan-2025 RA prominence floor. Both the persistent footer and the per-answer disclaimer render at the Small 12px tier, so both surfaces clear the regulator minimum identically. Persistent footer is sticky on mobile (always above the fold). Per-answer footer renders inline with every answer (always next to the content it qualifies). Verified.

## Information Architecture (Cited-Q&A Page)

Order on the single Phase 1 home-page-as-chat surface, top to bottom on mobile:

1. **Hero block — expanded state** (renders only on first session entry, before any question has been asked):
   - Display 28px heading: `Ask DRHPLens about Swiggy.`
   - Body 16px subheading: `One Indian IPO. One prospectus. Ask anything. Every answer cites the page it came from.`
   - Spacing: `xl` (32px) below the subheading before the DRHP metadata header begins.
1a. **Hero block — collapsed state** (replaces the expanded hero the moment the user submits their first question, persists for the rest of the session):
   - A single line in Body 16px/400 left-aligned, neutral text color (`#0F172A`): `Ask DRHPLens about Swiggy.`
   - Spacing: `md` (16px) below this line before the DRHP metadata header begins.
   - The expanded subheading and the 28px Display heading are removed from the DOM (not just visually hidden) so the metadata header and chat history sit higher on the viewport.
   - Transition: no animation in Phase 1 — instant swap on first message submit (a half-second crossfade is a `prefers-reduced-motion`-respecting Phase 6 polish item, not a Phase 1 deliverable).
2. **DRHP metadata header** (Surface-secondary `#F4F5F7` background, `lg` padding, `8px` radius):
   - Single line on desktop, wrapping rows on mobile
   - Label-value pairs (label Body 16px/600 muted; value Body 16px/600 primary): `Issuer · Swiggy Limited` · `Issue size · ₹11,327 cr` · `Listed · Nov 2024` · `DRHP source · SEBI` (the last is a link to the canonical SEBI-hosted PDF)
   - Lakh/crore notation already in Phase 1 for issue size only (low-effort, matches mental model; full UI-04 lands in Phase 4)
3. **Chat history** (renders only after first question):
   - Each turn: question (right-aligned Body 16px/400 in a `Surface-secondary` rounded bubble) → answer (left-aligned Body 16px/400 prose with inline citation chips, citation expanders, per-answer disclaimer)
   - History is session-scoped only — no persistence between page refreshes in Phase 1
4. **Question input** (Streamlit `st.chat_input`):
   - Sticky at bottom on mobile (above the persistent footer)
   - Placeholder text from Copywriting Contract
   - On focus: 2px accent-color ring
5. **Empty state** (renders in the chat-history region before first question):
   - Heading 20px/600: `Nothing asked yet.`
   - Three suggestion chips (label Body 16px/600, Surface-secondary background) showing example questions — clicking fills the input, does not auto-submit
6. **Persistent footer** (fixed bottom, contract above)

**Hidden until needed:** `/methodology` stub link is only in the footer (not in a header nav) — a header nav for a single-page app is bloat.

**Above-the-fold priority on mobile (375px viewport):** Hero heading + DRHP metadata header + question input. Disclaimer footer is sticky-visible. Chat history scrolls below.

## Accessibility Contract (WCAG 2.1 AA minimum)

| Concern | Implementation |
|---------|----------------|
| Keyboard navigation | All citation chips are `tabindex="0"` with `role="button"` and `aria-describedby` pointing at their expander region. `Enter` and `Space` open the expander. `Tab` moves between chips in document order. |
| Screen reader | Citation chips announce as: `link, citation 1, opens source-text panel` via `aria-label` derived from the chip number. Source-text panel announces verbatim DRHP text plus the "View DRHP page N on SEBI" link. The per-answer disclaimer is wrapped in `<small aria-label="Informational disclaimer">`. |
| Color contrast | All text combinations verified AAA above. No color used as the only differentiator of state (refusal banner has left-border + heading text, not just background color). |
| Focus indicators | 2px accent-color outline on all focusable elements, 2px offset from element edge. Never `outline: none` without a replacement. |
| Reduced motion | If `prefers-reduced-motion: reduce`, the expander opens instantly (no slide animation) and the loading-state's optional pulse animation is suppressed. |

## Streamlit-Specific Constraints

What's achievable with stock Streamlit 1.36+ widgets vs. what needs HTML/CSS injection vs. what needs custom components:

| Surface | Implementation path |
|---------|---------------------|
| Question input | `st.chat_input` (stock, sticky-bottom on mobile by default) |
| Chat history | `st.chat_message` with `role="user"` / `role="assistant"` containers (stock) |
| Citation chips inline in answer prose | `st.markdown(answer_with_inline_html, unsafe_allow_html=True)` where the answer string includes `<sup class="drhp-cite" ...>[1]</sup>` markup |
| Citation expanders below answer | `st.expander(f"[{n}] DRHP page {page} · {section}", expanded=False)` — one expander per deduped chip, ordered to match in-text appearance |
| Refusal banner | `st.markdown(refusal_html, unsafe_allow_html=True)` with a styled `<div>` — NOT `st.warning` (`st.warning` uses yellow alert chrome that reads as system-error, not content) |
| First-use modal | `st.dialog` (Streamlit 1.32+; verify version pin in plan-phase). If `st.dialog` unavailable, fallback to a session-state-gated `st.container` with a backdrop overlay via CSS injection. |
| Persistent footer | CSS-injected `<div>` with `position: sticky` (mobile) / `position: relative` (desktop) at the foot of the main column. Streamlit's hidden default footer must be hidden via CSS: `[data-testid="stFooter"] { display: none; }` |
| Per-answer disclaimer | Embedded as the closing line of the assistant's markdown payload, styled via the same CSS rules as the persistent footer's typography |
| Reformulation suggestion chips | `st.columns(3)` with `st.button` per chip, styled via CSS targeting `[data-testid="stButton"] button` — clicking the button sets `st.session_state.draft_question` which pre-fills `st.chat_input` on the next rerun |
| Loading state | `st.status` container (Streamlit 1.28+) with `state="running"` while LLM is in flight, transitioning to `state="complete"` on success or being replaced by the refusal banner on failure |
