Where a value is unchanged from Phase 1, this contract states "inherits Phase 1 §X" rather than re-deriving it.

## Typography

| Role | Size | Weight | Line Height | NEW Phase-2 usage (inherited roles only) |
|------|------|--------|-------------|-------------------------------------------|
| Small | 12px | 400 (regular); italic variant | 1.5 | Card metadata labels (sector · listing date · issue size); financials table column headers; "risk N of M" counter; per-block per-answer disclaimer; "Not disclosed" note; split-bar percentage captions; breadcrumb |
| Body | 16px | 400 (regular); 600 (semibold) for emphasis/labels | 1.55 | Business-summary prose; risk-cluster prose; use-of-proceeds line items; promoter field label-value pairs (label 600); card issuer name (600); financials table data cells |
| Heading | 20px | 600 (semibold) | 1.3 | Each of the 6 snapshot field-block headings ("Business", "Key Financials", "Risk Factors", "Use of Proceeds", "Promoters & Management"); catalogue page heading; empty/error-state headings |

**Rules (inherited + Phase-2 additions):**
- No font sizes outside the four declared roles. No new weights. (Hard checker gate — same as Phase 1.)

## Color

**Inherits Phase 1 §Color verbatim. 60/30/10 split unchanged. The no-green/red invariant is Phase 2's single hardest constraint and is re-asserted below for the new signal surfaces.**

| Role | Value | NEW Phase-2 usage |
|------|-------|-------------------|
| Dominant (60%) | `#FFFFFF` | Catalogue page background; snapshot page background; IPO card background; snapshot field-block background |
| Surface-secondary (30%) | `#F4F5F7` | Snapshot metadata-header block background (reuses Phase 1 `.drhp-metadata-header`); financials table zebra/header row; **the "fresh issue" segment fill of the OFS-vs-fresh split bar**; card hover background |
| Text-primary | `#0F172A` | All headings, prose, card issuer names, table data |
| Text-muted | `#475569` | Card metadata labels; table headers; "risk N of M"; per-block disclaimer; "Not disclosed" note; **the OFS segment fill border/label of the split bar** |
| Border-subtle | `#E2E8F0` | IPO card border (1px); snapshot block divider; financials table gridlines; split-bar outline; breadcrumb separator |
| Accent (10%) | `#1E40AF` (slate-indigo) | Citation chips inside blocks (inherited); card focus ring; snapshot-block expander focus ring; breadcrumb link; **the "OFS (promoter sells)" segment of the split bar** (see split-bar contract); the back-to-catalogue link |
| Destructive | not used | No destructive actions in Phase 2 (browsing + reading only). Reserved hex `#B91C1C` remains unused. |

**Accent reserved for (explicit list — never "all interactive elements"):**
1. Citation chip background + chip hover/focus ring (inherited, unchanged)
2. IPO card keyboard-focus ring + snapshot-block-link focus ring
3. Breadcrumb / back-to-catalogue navigation links
4. The **OFS ("promoter sells existing shares") segment** of the OFS-vs-fresh split bar — the one place accent carries semantic weight in Phase 2 (see split-bar contract for why this is honesty-neutral, not a "bad" signal)
5. The methodology link in the persistent footer (inherited)

**Explicitly NOT colored with accent:** card issuer names, all snapshot prose, financials numbers, promoter fields, the "fresh issue" segment of the split bar, "Not disclosed" notes, risk text.

**The split bar is the ONLY place a number is encoded by color, and it is deliberately neutral:**
- OFS segment = **accent `#1E40AF`** (slate-indigo). Fresh-issue segment = **Surface-secondary `#F4F5F7`** with a `#475569` label.
- This is the SAME accent already used for citation chips — it reads as "the system is pointing at this," NOT as "this is bad." There is **no red, no warning color, no green**. A 100%-OFS issue and a 100%-fresh issue use the identical two-tone palette; only the proportion and the printed percentages differ. The signal is the *proportion the user reads*, not a color verdict. (D2-06, ROADMAP SC-5, honesty-first invariant.)
- Rationale recorded for the checker: encoding OFS in accent rather than red avoids implying "OFS = bad / fresh = good." OFS is a neutral disclosed fact; many strong IPOs are partial-OFS. The product describes; it does not judge.

**Contrast verification (planner re-checks after CSS):**
- Card metadata `#475569` on `#FFFFFF`: 7.1:1 (AAA) — inherited.
- Split-bar OFS label: white text on `#1E40AF`: 8.6:1 (AAA) — same as chip.
- Split-bar fresh-issue label `#475569` on `#F4F5F7`: ≥ 6.5:1 (AAA at this size) — planner verifies exact ratio.
- Financials table data `#0F172A` on `#FFFFFF` and on `#F4F5F7` zebra rows: ≥ 7:1 (AAA).
- Split bar carries a **text percentage label inside/under each segment**, so color is never the sole differentiator (WCAG 1.4.1 — inherited rule).

## Information Architecture — Catalogue Page (NEW)

**Route:** `/` becomes the catalogue (the new landing). Resolves objective dimension 6: **the catalogue is the new landing page.** The Phase 1 single-IPO chat home is no longer the root — its chat surface moves onto each snapshot page (P2-L5). Existing Phase 1 home logic is generalized to be parameterized by `drhp_id`.

> **FLAG-ROUTING** (planner): Phase 1's `app.py` IS the root `/` and hard-codes Swiggy. Phase 2 must turn `app.py` into the catalogue landing and move the chat+metadata+answer rendering into the snapshot page, parameterized by `drhp_id`. The Swiggy deep-link must not break — `/snapshot?drhp_id=swiggy` (or the chosen multipage equivalent) must resolve. See §Streamlit Routing.

Top-to-bottom on mobile:

1. **Catalogue hero** (Display 28px + Body 16px subheading):
   - Heading: see Copywriting Contract (`Indian IPOs, read honestly.`)
   - Subheading: one line describing the catalogue.
   - Spacing: `xl` (32px) below subheading before the grid.
2. **IPO card grid** (`.drhp-catalogue-grid`):
   - Desktop (≥1024px): 3 columns. Tablet (640–1023px): 2 columns. Mobile (<640px): **single column** (inherits Phase 1 mobile-first single-column default).
   - Cards ordered most-recent-listing-first; currently-open IPOs (1–2 per OPS-01) sort to the top with a neutral text tag `Open now` (Small 12px muted — text only, no color, no badge fill).
   - Card grid implemented via `st.columns(n)` with one `.drhp-ipo-card` per cell.
3. **Persistent footer** (inherited, sticky on mobile).

**IPO card contents (`.drhp-ipo-card`) — factual only, NO performance badges (P2-L1):**
- Row 1: **Issuer name** — Body 16px / 600, Text-primary.
- Row 2: **Sector** — Small 12px / 400 muted (text label, e.g. `Food delivery`).
- Row 3: **Listing date** · **Issue size** — Small 12px / 400 muted, middot-separated (e.g. `Listed Nov 2024 · ₹11,327 cr`). Currently-open IPOs show `Open · {date}–{date} · ₹X cr`.
- The entire card is one clickable/tappable region → snapshot page for that `drhp_id`.
- **What a card MUST NOT contain:** no listing-gain %, no green up-arrow / red down-arrow, no "hot"/"avoid" tag, no star rating, no GMP (GMP is Phase 4 and read-only-with-caveats even then). A winner and a flop look identical in chrome.

**Card states:**
- **Default:** `#FFFFFF` background, 1px `#E2E8F0` border, `8px` radius, `lg` padding.
- **Hover (desktop):** background → `#F4F5F7` (Surface-secondary); border unchanged; cursor pointer. No transform/shadow that implies value.
- **Focus (keyboard):** 2px `#1E40AF` outline, 2px offset (inherited focus-ring rule).
- **Active/tap:** background `#F4F5F7`; navigates on release.
- No "visited" styling — catalogue cards are navigational but session-neutral.

## Information Architecture — Snapshot Page (NEW)

**Route:** `/snapshot?drhp_id={id}` (or Streamlit multipage equivalent — see §Routing). One page per IPO, rendered from the offline-precomputed cache keyed by `drhp_id` (P2-L4).

Top-to-bottom on mobile (this is the locked block ORDER — resolves objective dimension 2):

1. **Breadcrumb / back link** (`.drhp-breadcrumb`): `← All IPOs` (accent link) `/ {Issuer}` (muted). Small 12px. Tap target ≥44px. Spacing `lg` below.
2. **Page title** — Display 28px / 600: the **issuer name** (e.g. `Swiggy Limited`). Used once.
3. **Block 1 — Metadata header (SNAP-02)** — reuses Phase 1 `.drhp-metadata-header` (Surface-secondary `#F4F5F7`, `lg` padding, 8px radius). Label-value pairs (label Body 16/600 muted, value Body 16/600 primary): `Price band` · `Lot size` · `Issue dates` · `Issue size` · `Fresh vs OFS` (summary % — full bar appears in Block 5) · `Lead managers` · `DRHP source` (link to SEBI/BSE/NSE PDF). Each value that came from the RHP cover page carries a citation chip.
4. **Block 2 — Business summary (SNAP-03)** — `.drhp-snapshot-block`. Heading 20px `Business`. Plain-English prose (Body 16px) with inline citation chips + expanders (Phase 1 renderer, unchanged). Per-block per-answer disclaimer at the foot.
5. **Block 3 — Key Financials (SNAP-04)** — Heading 20px `Key Financials`. 3–5 year table (`.drhp-fin-table`): rows = Revenue, Profit/(Loss), Margin %, Debt, ROE, ROCE; columns = fiscal years (FY22–FY24…). `tabular-nums`. Each metric row carries a chip to the DRHP restated-financials page it came from. See §Financials Rendering.
6. **Block 4 — Risk Factors (SNAP-05)** — Heading 20px `Risk Factors`. Prioritized list of risk clusters, each with a `Risk N of M` counter and its own citation to the original DRHP risk text. See §Risk Block.
7. **Block 5 — Use of Proceeds (SNAP-06)** — Heading 20px `Use of Proceeds`. **The OFS-vs-fresh split bar is the FIRST element in this block, foregrounded above the line-item breakdown.** Then the fresh-issue use-of-proceeds line items (Body 16px, each cited). See §Split-Bar Contract.
8. **Block 6 — Promoters & Management (SNAP-07)** — Heading 20px `Promoters & Management`. Label-value pairs: promoter names, pre/post-issue holding %, pledging status, prior matters — each cited.
9. **Q&A chat (Phase 1, inherited)** — separated from Block 6 by a `2xl` (48px) gap and a 1px `#E2E8F0` divider. **Placement: BELOW the blocks, not a tab** (resolves objective dimension 2). Rationale: snapshot is the "read" surface; chat is the "drill-in" surface (D2-08). A tab would hide the chat and break the "read then ask" flow; co-locating below keeps both on one scroll. The chat reuses the Phase 1 hero-collapsed line, `st.chat_input`, chat history, citation chips, refusal banner — all bound to THIS page's `drhp_id`.
10. **Persistent footer** (inherited, sticky on mobile).

**Citations inside every block:** identical to Phase 1 Q&A. Each block's cached `GroundedAnswer` runs through `ui/chip.py` → superscript `[N]` chips → `ui/expander.py` → one `st.expander` per chip showing the verbatim DRHP span + `View DRHP page N →` link + `claim_id` metadata footer (P2-L4). **No new citation UI is invented** — snapshot fields cite exactly like answers.

**Snapshot block CSS (`.drhp-snapshot-block`):** `#FFFFFF` background, 1px `#E2E8F0` top divider OR `8px`-radius bordered card (planner picks one and applies it to all six identically), `lg` padding, `xl` vertical gap between blocks. Heading uses `.drhp-snapshot-block-heading` (Heading 20px/600).

## Visuals — OFS-vs-Fresh Split Bar Contract (THE Phase 2 signal element)

The single most important new visual. SNAP-06 / D2-06 / ROADMAP SC-5. Renders at the top of Block 5 (Use of Proceeds).

**Structure (`.drhp-split-bar`), top to bottom:**
1. **Caption row** (Small 12px muted): `Of the total issue, X% is a fresh issue (money to the company) and Y% is an Offer for Sale (existing shareholders sell).` — plain-English, describes both sides neutrally. Carries a citation chip to the RHP cover/capital-structure page.
2. **The bar itself** — a single horizontal bar, full content width, height 28px, `8px` radius, 1px `#E2E8F0` outline:
   - **OFS segment:** width = OFS %, fill `#1E40AF` (accent), white inline label `OFS Y%` (Small 12px/600) when the segment is wide enough; otherwise the label moves to the caption.
   - **Fresh segment:** width = fresh %, fill `#F4F5F7` (Surface-secondary), `#475569` inline label `Fresh X%`.
   - Segments sit flush; the boundary is a 1px `#E2E8F0` divider.
3. **Percentage labels** render as TEXT inside/beside each segment (never color-only — WCAG 1.4.1).

**Honesty rules (checker-enforced):**
- No red, no green, no amber. OFS is accent (the "system points here" color), fresh is neutral grey. A 90%-OFS issue is NOT styled as a warning; it shows a wide accent segment and the printed `OFS 90%` — the user draws the inference, the product does not.
- The bar must be readable as raw proportion: a user seeing the bar without labels still perceives the ratio, but labels are always present for accessibility.
- 0%-OFS (pure fresh issue) renders a full-width grey bar labeled `Fresh 100% · No OFS` — explicitly stating there is no promoter cash-out, as a positive-by-omission fact, still in neutral grey (no green).
- 100%-OFS renders a full-width accent bar labeled `OFS 100% · No fresh capital raised`.

**Mobile:** bar goes full-bleed within content padding; caption wraps; labels stay inside segments if ≥20% width else drop to the caption line. Bar height stays 28px (the bar is not interactive, so the 44px target rule does not apply to the bar; the citation chip in the caption keeps its 44px tap target).

**Accessibility:** the bar is wrapped in `role="img"` with an `aria-label`: `Issue split: OFS Y percent, fresh issue X percent.` so screen readers get the proportion without parsing visual segments.

## Visuals — Financials Snapshot Rendering (SNAP-04)

**Decision: a TABLE, not sparklines.** (Resolves objective dimension 4.) Rationale: 3–5 discrete fiscal years of 6 metrics is tabular data; a table with `tabular-nums` is more honest and more accessible than a sparkline (which implies a trend read the product is not yet making — trend interpretation is closer to Phase 5's forecasting posture and would risk an implied verdict). Sparklines deferred; not introduced in Phase 2.

**`.drhp-fin-table`:**
- Columns: fiscal years left-to-right oldest→newest (e.g. `FY22 · FY23 · FY24`). Column headers Small 12px/600 muted on a `#F4F5F7` header row.
- Rows: `Revenue`, `Profit / (Loss)`, `Margin %`, `Debt`, `ROE`, `ROCE`. Row labels Body 16px/600 muted (left column, sticky on horizontal scroll).
- Data cells: Body 16px/400 primary, `font-variant-numeric: tabular-nums`, right-aligned.
- **Negatives:** a loss renders as `(₹1,234 cr)` in parentheses — the Indian/accounting convention — **in the same `#0F172A` text color as a profit. NO red for losses.** This is the honesty-first invariant applied to numbers: a loss is a disclosed fact shown identically to a profit; only the parentheses and value differ. (Checker gate.)
- Magnitudes in lakh/crore (P2-L6). `%` metrics to one decimal.
- Zebra striping via `#F4F5F7` on alternate rows (Surface-secondary) for scan-ability — no color coding of values.
- **Each metric row carries a citation chip** to the DRHP restated-financials page it was extracted from.
- **Missing year/metric:** the cell shows `—` (em dash) with a `title`/`aria-label` of `Not disclosed in this DRHP`; never a zero, never an interpolated guess (P2-L8).

**Mobile:** table becomes horizontally scrollable inside the block (`overflow-x: auto`), row-label column sticky-left so the user always sees which metric they're reading. This is the one place horizontal scroll is permitted; everything else stays single-column.

## Visuals — Risk-Factors Block (SNAP-05)

Prioritized list of risk clusters. Each cluster is a discrete `.drhp-risk-item`.

**Per risk item, top to bottom:**
1. **Counter** (Small 12px/400 muted): `Risk N of M` — makes the prioritization and the total count explicit (resolves objective dimension 5: "this is risk N of M").
2. **Risk summary** (Body 16px/400 prose) — plain-English cluster summary with inline citation chip(s) pointing to the original DRHP risk text. The chip expander shows the verbatim risk-factor paragraph (Phase 1 expander, unchanged).
3. Items separated by `md` (16px) gap; no dividers needed (whitespace separates).

**Honesty rules:**
- Prioritization order is by the offline extractor's salience ranking; the order itself is the only "weighting" signal — **no severity color, no red flag icon, no risk-score badge.** (Phase 3 EXTRACT-02 introduces *visible confidence scores* and the structured red-flag table; Phase 2 does NOT pre-empt that — it shows prioritized prose clusters only.)
- The `Risk N of M` counter is descriptive, not evaluative — it says "this is the Nth most prominent risk we surfaced," not "this is the Nth most dangerous."
- If the DRHP risk section failed to extract, the block shows the standard `Not disclosed in this DRHP.` honesty note (P2-L8), not an empty list rendered as "no risks."

## Cross-Cutting Invariants (re-stated for Phase 2)

1. **No green/red coding** — re-asserted as Phase 2's hardest constraint. Catalogue cards for winners and flops are visually identical; financial losses render in normal text with parentheses; the OFS split bar uses accent + neutral grey, never red/green. (D2-07, ROADMAP cross-cutting, PITFALL P21 precedent.)
2. **No performance badges, no GMP, no verdict.** The catalogue and snapshot describe; they never rank, rate, or recommend. GMP is Phase 4 and read-only-with-caveats even then.
3. **Span-level citations only, from the same `claim_id` schema.** Snapshot field blocks render citations through the unchanged Phase 1 chip+expander path. Phase 3 METHOD-01 consumes the same structure — keep it stable.
4. **Absence shown as absence.** A failed extraction shows `Not disclosed in this DRHP.` / `—`, never a fabricated or zero-as-real value (P2-L8 — the Phase 2 anti-hallucination surface).
5. **Disclaimer surfaces non-negotiable.** Persistent footer on catalogue + snapshot; per-answer footer below every cited block and every chat answer. (Inherited; absence is a release-blocker.)
6. **Banned-token scrubber is hard-block-and-regenerate** — applies to all snapshot LLM copy (business summaries, risk clusters) and all Phase-2 static copy (import-time assert in `ui/copy.py`).
7. **Mobile-first.** Every contract above states mobile behaviour first. Snapshot stays single-column at every breakpoint; only the card grid and financials table adapt horizontally.
8. **Extend, don't restyle.** No inherited token, type size, color, or component CSS is redefined. Phase 2 adds new classes that consume the inherited `--drhp-*` custom properties.
