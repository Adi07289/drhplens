<objective>
Snapshot pre-computation, per RESEARCH §Pattern 3. For each `drhp_id`, run the EXISTING
compiled agent 6× with 6 canned queries (metadata / business / financials / risks /
use-of-proceeds / promoter) and serialize each resulting `GroundedAnswer` (claim_ids and
all) into `data/snapshots/<drhp_id>.json`. When the agent refuses a field (DRHP silent),
store the `RefusalResponse` instead — the honesty-first invariant carried into snapshots
(critical for SNAP-07 pledging). Compute the OFS-vs-fresh % for SNAP-06. Scrub every
answer_prose at pre-compute time so banned tokens can never reach a committed snapshot.

NO new LLM path, NO "snapshot mode" in the graph (RESEARCH anti-pattern). The snapshot
record IS a serialized GroundedAnswer, so Wave 4's UI reuses the Phase 1 chip renderer for free.

CODE-NOW-DEFER: write precompute() + schema + scrubber-gate + OFS/fresh logic + unit tests
now (monkeypatch GRAPH.invoke to return canned GroundedAnswer/RefusalResponse — no live LLM
or Qdrant needed). The real 6×8 pre-compute run against live Qdrant+LLM is deferred to Plan
06's runbook; a committed Swiggy snapshot.json (built from monkeypatched fixtures here, or
hand-built from Phase 1's known-good answers) seeds Wave 4's UI work.

Purpose: deliver SNAP-02..07 data layer — every snapshot field cited via the reused
GroundedAnswer schema + honest refusals + neutral OFS/fresh split.
